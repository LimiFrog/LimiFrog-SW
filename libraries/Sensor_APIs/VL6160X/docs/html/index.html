Documentation

Introduction

This document describes the API functions for VL6180 products familly to aid the development of applications. Although this User Manual is the central reference document to use when programming, you may first read the VL6180X basic ranging application notes AN4545 and DT0030 for Ambient light sensing applications at www.st.com/vl6180x/

Some of the API functions are platform-dependent (specially I2C access) so need to be adapted to the platform used by the customer.

API Architecture

API delivery is organized as follows:

API source code is structured (conceptually) in several modules defined as follows :

  • API Low level functions (described in API Low Level Functions module of this document) : Performs unitary operations through basic register access. Low level functions are implemented in the core directory of the API delivery. Most of the time low level functions should not be called directly by the application or should be reserved for an advanced usage of the API.
  • API High level functions (described in API High Level Functions module of this document) : Calls low level API functions, to carry out measurements for the application. By construction, these high level functions are mainly working in polling mode in opposition to the interrupt mode which can be handled only at the application level. High level functions are also implemented in the core directory.
  • Configuration (described in Configuration module of this document) : This module gives the configuration of the API for the targeted application and product. All settings defined in this configuration module are guaranteed by ST. Any changes in the API configuration could be functional but are not guaranteed by ST. Configuration module is implemented in the config directory of the API delivery.
  • Platform (described in Platform module of this document) : This module must be adapted for the platform on which API is running (types, debug, logging, I2C access). Platform module is implemented in the platform directory.

API Use Example

This code example shows how API functions can be used by the application to perform a basic ranging operation (using default settings of the API). All variables and functions prefixed by VL6180X are defined in the API :

void Sample_SimpleRanging(void) {
VL6180xDev_t myDev;
VL6180x_RangeData_t Range;
MyDev_Init(myDev); // your code init device variable
MyDev_SetChipEnable(myDev); // your code assert chip enable
MyDev_uSleep(1000); // your code sleep at least 1 msec
VL6180x_InitData(myDev);
VL6180x_Prepare(myDev);
do {
VL6180x_RangePollMeasurement(myDev, &Range);
if (Range.errorStatus == 0 )
MyDev_ShowRange(myDev, Range.range_mm,0); // your code display range in mm
else
MyDev_ShowErr(myDev, Range.errorStatus); // your code display error code
} while (!MyDev_UserSayStop(myDev)); // your code to stop looping
}

The above example relies on the VL6180x_RangePollMeasurement() API function which is a blocking function (This will start a single shot measurement and function will wait/poll until a distance measurement is available). A better usage of the CPU could be done using the VL6180x_RangeGetMeasurementIfReady() API function which will only check if a distance measurement is available (non-blocking function). Of course, it is up to the CPU to start (and re-start) ranging operations using the VL6180x_RangeStartSingleShot() function. See how this free-running mode can be implemented in this example:

void Sample_FreeRunningRanging(void) {
VL6180xDev_t myDev;
VL6180x_RangeData_t Range;
int status;
MyDev_Init(myDev); // your code init device variable
MyDev_SetChipEnable(myDev); // your code assert chip enable
MyDev_uSleep(1000); // your code sleep at least 1 msec
VL6180x_InitData(myDev);
VL6180x_Prepare(myDev);
VL6180x_RangeClearInterrupt(myDev); // make sure no interrupt is pending
/* kick off the first measurement */
VL6180x_RangeStartSingleShot(myDev);
do {
// TODO add your code anything in a loop way
VL6180x_PollDelay(dev); // simply run default API poll delay that handle display in demo
// check for range measure availability
status= VL6180x_RangeGetMeasurementIfReady(myDev, &Range);
if( status == 0 ){
// Application must check Range.errorStatus before accessing the other data
// If Range.errorStatus is DataNotReady, application knows that it has to wait a bit before getting a new data
// If Range.errorStatus is 0, application knows it is a valid distance
// If Range.errorStatus is not 0, application knows that reported distance is invalid so may take some decisions depending on the errorStatus
if (Range.errorStatus == DataNotReady)
continue;
if (Range.errorStatus == 0)
MyDev_ShowRange(myDev, Range.range_mm,0); // your code display range in mm
else
MyDev_ShowErr(myDev, Range.errorStatus); // your code display error code
/* re-arm next measurement */
VL6180x_RangeStartSingleShot(myDev);
}
else{
// it is an critical error
HandleError("critical error on VL6180x_RangeCheckAndGetMeasurement");
}
} while (!MyDev_UserSayStop(myDev)); // your code to stop looping
}

API Configuration

The API static configuration is defined in vl6180x_cfg.h file (in config directory). All features defined in this file are guaranteed by ST. If user modifies the file, this is done at his own risk. Configuration settings are described in the Configuration module of this document...

API Porting

The API is delivered as open source code and has been validated on various ST platforms including Nucleo boards. API contains few files to be adapted and potentially few functions to be implemented to adapt to the end-user platform.

Here are the steps you need to follow to port API on your platform :

1. VL6180x device type definition

User must provide VL6180xDev_t type (in vl6180x_platform.h file) as all API functions and macros rely on VL6180xDev_t dev (given as first argument). This dev object does the link between API and platform abstraction layer and is passed from function to function down to final platform abstraction layer that handles final access to the device :

int VL6180x_xxxx(VL6180xDev_t dev, ... )

In single device case, dev can be as simple as an integer being the i2c device address

typedef uint8_t VL6180xDev_t; /* simplest single device example "dev" is the i2c device address in the platform AL*/

For more elaborated platform, dev can be a pointer to a structure containing all necessary items for the platform.

struct MyDev_t {
struct VL6180xDevData_t Data;
//TODO ADD HERE any extra device data
int i2c_bus_num;
int i2c_dev_addr;
mutex_t dev_lock ;
int i2c_file;
};
typedef struct MyDev_t *VL6180xDev_t;
#define VL6180xDevDataGet(dev, field) (dev->Data.field)
#define VL6180xDevDataSet(dev, field, data) (dev->Data.field)=(data)

2. I2C read & write access

API low-level functions rely on a few set of read & write functions which perform I2C access to the device. These functions must be implemented with respect to the platform on which API is compiled and running. You may go an see in API Register access functions module for more details on these functions :

  • VL6180x_WrByte()
  • VL6180x_WrWord()
  • VL6180x_WrDWord()
  • VL6180x_UpdateByte()
  • VL6180x_RdByte()
  • VL6180x_RdWord()
  • VL6180x_RdDWord()
  • VL6180x_RdMulti()

A default/optional implementation of these functions is provided in the platform/cci_i2c directory. If user decides to use this implementation, only 2 functions remain to be implemented : VL6180x_I2CRead() and VL6180x_I2CWrite(). Go and see the CCI to RAW I2C translation layer module to get more details on these functions

Here is a code example for STM32 compliant with the STM32Cube Hardware Abstraction Layer:

int VL6180_I2CWrite(uint8_t addr, void *buff, uint8_t len)
{
int status;
status = HAL_I2C_Master_Transmit(i2c_bus, addr, buff, len , def_i2c_time_out);
return status;
}

The function VL6180x_RdMulti() is optional but is required if VL6180x_HAVE_MULTI_READ macro is defined in vl6180x_platform.h file.

3. Data Types declaration

API functions rely on data types which are defined in vl6180x_types.h file (under platform/template directory). This file may require user attention and porting in case of warning messages. See Basic type definition module for more details...

4. Delay for polling operations

API polling high level functions (VL6180x_RangePollMeasurement() and VL6180x_AlsPollMeasurement()) do call the function VL6180x_PollDelay() inside their while loop. The default implementation of the VL6180x_PollDelay() function does nothing (see VL6180x_PollDelay macro in vl6180x_platform.h file). You may decide to delete this macro and implement your own VL6180x_PollDelay() function.

5. API logging

All API functions entry and leave can be logged to help debugging issues. By default logging is disabled (see VL6180X_LOG_ENABLE macro in vl6180x_platform.h). If logging is enabled, a small set of macros must be implemented to adapt logging operation to the platform : LOG_FUNCTION_START, LOG_FUNCTION_END and LOG_FUNCTION_END_FMT
